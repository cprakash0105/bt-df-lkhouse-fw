Config-Driven Lakehouse Framework

GCP Native · Apache Iceberg · BigQuery · PySpark
Chandra Prakash · BT Group · June 2026

What Is This Framework?

  • A config-driven data pipeline framework on GCP
  • Add a new dataset by dropping a YAML file — zero code changes
  • Handles: ingestion, DQ, dedup, schema evolution, SCD, consumption
  • Fully automated via Cloud Functions (event-driven)
  • Open formats: Apache Iceberg + BigQuery

Think of it as: "Infrastructure from metadata"

Architecture

Config-Driven: Add a Table

# config/tables/cibil_bureau_feed.yaml

table: cibil_bureau_feed
source: landing/cibil_bureau_feed
primary_key: customer_id
dedup_order_by: ingestion_ts DESC

dq_rules:
  not_null: [customer_id, cibil_score]
  range:
    cibil_score: [300, 900]
  format:
    pan_number: pan

schema_evolution:
  allowed: [add_column, type_widen]
  blocked: [drop_column, type_narrow]

pii_fields: [pan_number, mobile_number, email_address]

Drop this YAML in GCS → pipeline processes the table automatically. No code.

Each engine reads from the same YAML config. Single source of truth.

Ingest: Landing → Reservoir

# What ingest.py does:
1. Read JSONL from gs://bucket/landing/{table}/
2. Add ingestion_ts = current_timestamp()
3. Infer schema from data (or use config overrides)
4. Write as Parquet to gs://bucket/reservoir/{table}/
5. Log to pipeline_monitor: records_in count
  • Append mode — never overwrites history
  • Type overrides from config (e.g., amount → DOUBLE)
  • Handles schema-on-read: new columns just appear

Curate: Reservoir → CCN

# What curate.py does:
1. Read Parquet from reservoir
2. DQ Validation:
   - not_null: reject rows where required fields are null
   - range: reject rows outside bounds
   - format: reject rows failing regex (PAN, email, etc.)
   - accepted_values: reject rows with invalid codes
   - positive: reject rows with negative amounts
3. Deduplication:
   - Window function on primary_key
   - ORDER BY dedup_order_by (latest wins)
4. Schema Evolution:
   - Detect drift (new/changed/dropped columns)
   - Check governance rules → allow or block
5. Write to Iceberg (merge-schema mode)
6. Log: records_in, records_out, records_rejected

Schema Evolution Governance

Incoming Data               Existing Iceberg Table
     ↓                              ↓

│  • New columns?    → add_column             │
│  • Type widened?   → type_widen             │
│  • Type narrowed?  → type_narrow            │
│  • Column gone?    → drop_column            │
└─────────────────────┬───────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│  Config says:                               │
│    allowed: [add_column, type_widen]        │
│    blocked: [drop_column, type_narrow]      │
│                                             │
│    ✅ Allowed → ALTER TABLE + continue      │
│    🚫 Blocked → Pipeline FAILS + alert     │
└─────────────────────────────────────────────┘

Prevents breaking changes from silently reaching consumers.

Deduplication

# Config:
primary_key: customer_id
dedup_order_by: ingestion_ts DESC

# Execution (PySpark window function):
SELECT *, ROW_NUMBER() OVER (
    PARTITION BY customer_id
    ORDER BY ingestion_ts DESC
) AS _rn
FROM reservoir_data
WHERE _rn = 1
  • Latest record wins (configurable order)
  • Runs AFTER DQ — only valid records get deduped
  • Primary key from config (set by SD during discovery)

Consume: CCN → Data Product

# config/consumption/loan_eligibility_360.sql

CREATE OR REPLACE TABLE
  `bt-df-lkhouse.lakehouse_dataproduct.loan_eligibility_360` AS
SELECT
    c.customer_id, c.name, c.region,
    b.cibil_score,
    k.kyc_status,
    CASE
        WHEN b.cibil_score >= 750 AND k.kyc_status = 'verified'
            THEN 'pre_approved'
        WHEN b.cibil_score >= 650 AND k.kyc_status = 'verified'
            THEN 'eligible'
        ELSE 'not_eligible'
    END AS loan_eligibility_status
FROM `bt-df-lkhouse.lakehouse_ccn.customers` c
JOIN `bt-df-lkhouse.lakehouse_ccn.cibil_bureau_feed` b
    ON c.customer_id = b.customer_id
JOIN `bt-df-lkhouse.lakehouse_ccn.ekyc_provider_feed` k
    ON c.customer_id = k.customer_id

Drop SQL in config → pipeline materialises it. Consumer queries BigQuery directly.

Event-Driven Automation

YAML config lands in GCS (pushed by SD or manually)
        │
        ▼  GCS Eventarc trigger
Cloud Function (Gen2) fires:
        │
        ├── 1. Submit INGEST job → Dataproc cluster
        │       └── Wait for completion
        ├── 2. Submit CURATE job → Dataproc cluster
        │       └── DQ + dedup + schema evolution
        ├── 3. Create BQ external table (Iceberg via BLMS)
        ├── 4. Ensure consumption SQL exists
        ├── 5. Run CONSUME (materialise Data Product)
        ├── 6. Apply column-level security (Policy Tags)
        └── 7. Log everything to pipeline_monitor
        
Total time: ~3 minutes (1000 records)

Column-Level Security

# From config:
pii_fields: [pan_number, mobile_number, email_address]

# Framework automatically:
1. Creates Policy Tag taxonomy (if not exists)
2. Attaches PII policy tag to each column
3. Only users in "pii_reader" IAM group see values
4. Others get: ACCESS DENIED on those columns
  • One table, many policies — no masked copies
  • Access controlled at query time via IAM
  • PII fields identified by SD during discovery

Adding a New Dataset (Manual)

  1. Place source data in gs://bucket/landing/{name}/
  2. Create config/tables/{name}.yaml with DQ + schema rules
  3. Create config/consumption/{name}.sql for data product
  4. Upload config to GCS → Cloud Function auto-triggers
  5. Data appears in BigQuery within ~3 minutes

Or: let Semantic Discovery generate steps 2-4 automatically from a natural language description.

Adding a New Dataset (via SD)

  1. Steward: "I have a new CIBIL feed with customer_id, pan_number, cibil_score..."
  2. SD: profiles data, matches to BDEs, detects PII, suggests DQ
  3. Steward: clicks "Approve"
  4. SD auto-generates:
    • Pipeline YAML config (with inherited DQ from BDEs)
    • Data contract
    • Dataplex glossary entries + catalog links
    • SCD config (if applicable)
  5. Config pushed to GCS → pipeline runs → data in BQ

Total time: ~8 minutes (5 min human + 3 min pipeline)

Extensibility

  • New DQ rule type? Add handler in curate.py, reference in config YAML
  • New source format? Add reader in ingest.py (Avro, Parquet source, etc.)
  • New SCD type? Add logic in scd.py, reference in consumption YAML
  • New domain? Add BDEs to glossary → DQ rules auto-cascade to all tables
  • Cross-cloud? Add Trino for federation (planned)

Framework grows via config, not code changes.
