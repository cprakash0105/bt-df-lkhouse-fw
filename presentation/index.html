How Semantic Discovery Works

Inspired by Ab Initio's approach — but GCP-native, AI-enhanced, open.

Profile → Analyze → Fingerprint → Score → Suggest

Step 0: Auto-Profile

  • Fetches landing data from GCS automatically
  • Analyses each field: types, nulls, cardinality, patterns, ranges
  • Detects PII from actual values (not just names)
  • Identifies key candidates (>95% unique, <1% null)
  • Persists profile to gs://…/profiles/{dataset}/latest.json

No raw data leaves this step. Only statistical summaries proceed.

Steps 1–3: Multi-Layer Analysis

LayerMethodSpeedExample
1. Knowledge GraphSynonym lookupInstantcibil_score → "Credit Score"
2. Rules EngineDeterministic patternsInstant*_id → Identifier, *email* → PII
3. EmbeddingsSemantic similarity~100msloan_amount → "Transaction Amount"

Each layer only fires if previous layers couldn't resolve with high confidence.

Step 4: Fingerprinting

Compare field values against all 14 reference code sets in the glossary.

Field: "status"
Values: [open, in_progress, resolved, escalated, closed]

vs. order_statuses:      [delivered, shipped, processing...]  → 0% match ✗
vs. complaint_statuses:  [open, in_progress, resolved...]     → 100% match ✓
vs. payment_statuses:    [paid, overdue, partial, waived]     → 0% match ✗

Result: fingerprint = complaint_statuses (100%)
        → Override wrong BDE match

This is what fixed the customer_complaints DQ failure.

Step 5: Composite Confidence Score

Ab Initio synthesizes multiple tests. We do the same:

SignalWeightWhat it measures
Keyword30%Field name matches BDE synonym
Pattern25%Values match PII regex patterns
Fingerprint25%Values match reference code sets
Statistical20%Type + range + cardinality aligns with BDE

csat_score: keyword=100% + pattern=0% + fingerprint=0% + stat=80%
            → composite = 0.30×1 + 0.25×0 + 0.25×0 + 0.20×0.8 = 46%

complaint_id: keyword=100% + pattern=0% + fingerprint=0% + stat=70%
              → composite = 0.30×1 + 0.25×0 + 0.25×0 + 0.20×0.7 = 44%

The Role of the LLM

LLM Does (Reasoning)

  • Parse natural language → structured JSON
  • Generate SQL from requirements
  • Validate/correct suggestions
  • Generate business DQ from stats
  • Infer SCD type from intent

LLM Never Does (Execution)

  • Write to storage
  • Create/alter tables
  • Submit jobs
  • Modify schemas
  • Access raw data

Self-hosted (Gemma2 9B on Azure VM). Stateless. Isolated. Replaceable.
Only receives statistical summaries, never raw record data.

Data Boundary: What the LLM Sees

✗ NEVER sent to LLM:
  "CUST001", "ABCPD1234F", "750", "2024-01-15"

✓ ONLY sent to LLM:
  "cibil_score: integer, min=302, max=898, mean=701,
   null_pct=0.1%, distinct=587, distribution=normal"

The LLM reasons about what the data means, not what it contains.

Business-Level DQ Inheritance

"Define at conceptual level, push down — a handful of rules, not thousands."

Glossary BDE "Credit Score":
  dq_rules:
    not_null: true
    range: [300, 900]

→ Automatically applied to ALL tables using "Credit Score":
    • cibil_bureau_feed.cibil_score
    • credit_risk_model.bureau_score
    • loan_eligibility.score

= 1 rule defined. Applied everywhere. Zero engineering.

What SD Produces on Approval

  1. Pipeline Config (YAML) → pushed to GCS → Cloud Function auto-triggers
  2. Data Contract → schema, DQ SLAs, governance, consumers, masking
  3. BDE Terms → created in Dataplex Knowledge Catalog glossary
  4. Dataset Entry → registered in KC with EntryLinks to BA + BDEs
  5. SCD Config → dimension tracking (Types 1-6)
  6. Profile → persisted for audit trail and re-validation

All from one "approve" click. No manual config writing.

5 Design Principles

  1. Metadata is Executable — not documentation. BDE DQ → pipeline config → enforced.
  2. Contracts Before Code — every dataset has a contract. Pipeline compiled from it.
  3. AI Reasons, Determinism Executes — LLM suggests. Rules validate. Code executes.
  4. One Table, Many Policies — column security at query time, not storage time.
  5. Platform Learns — every approval enriches glossary for next onboarding.
